# MiniPrinter 固件开发路线

## 1. 当前基线

- 仓库已经是一个可用的 PlatformIO 工程。
- 当前 `platformio.ini` 目标板卡为 `esp32-s3-devkitc-1`，框架为 `arduino`。
- 代码侧目前仍是默认模板，尚未展开 RTOS、打印流程、驱动层与协议层。

## 2. 本阶段明确不做的事

- 不编写真实硬件驱动。
- 不创建打印头加热控制逻辑。
- 不修改 `platformio.ini` 配置，只把它作为当前工程状态的依据。

## 3. 建议开发路线

### Step 00. 仓库体检与文档落盘

- 目标：确认工程骨架、记录软件分层、建立引脚映射占位和进度文档。
- 产出：`docs/` 下的任务路线、架构说明、引脚表、进度记录。

### Step 01. 建立最小可维护目录骨架

- 目标：把 `src` 从单文件模板拆成可理解的分层结构。
- 建议优先建立：`app`、`tasks`、`rtos`、`common`、`config`。
- 说明：这一阶段先做“空心骨架”，不急着写真实外设逻辑，先把代码组织方式稳定下来。

### Step 02. RTOS 启动入口整理

- 目标：确定主入口、任务创建方式、任务间通信方式。
- 建议内容：
  - 明确 `setup()` 只负责最小初始化和任务启动。
  - 把业务循环迁移到 RTOS 任务。
  - 先选定一种基础通信机制，例如队列、事件组或消息结构体。

### Step 03. BSP 与配置层落地

- 目标：把板级资源和配置常量独立出来。
- 建议内容：
  - 建立引脚定义占位。
  - 建立电源、按键、故障脚、纸检测脚等板级抽象。
  - 对 GPIO 未知项统一标注 `TBD`，等原理图确认后再填。

### Step 04. 协议层与服务层骨架

- 目标：先把“收命令 -> 解释命令 -> 调用业务”的通路搭起来。
- 建议内容：
  - `protocol` 负责数据帧、命令码、解析状态机。
  - `services` 负责无硬件细节的打印服务、走纸服务、状态服务。
  - 先用模拟数据流验证，不直接落真实驱动。

### Step 05. 运动与状态管理

- 目标：逐步接入电机控制、纸张检测、按键输入、故障状态。
- 建议内容：
  - 先定义状态机，再决定驱动接口。
  - 真实芯片驱动前，先写接口与假实现，保证上层逻辑能单独测试。

### Step 06. 打印数据通路

- 目标：建立打印位图缓存、行缓冲、移位输出时序抽象。
- 重要限制：
  - 本阶段仍然可以只做“数据路径”，不做加热策略。
  - `DI`、`CLK`、`LAT`、`STB1~STB6` 等网络先抽象为接口，不直接下沉到真实波形控制。

### Step 07. 安全前提确认后再接入真实打印头控制

- 前提：原理图、供电能力、热敏头规格、安全策略都确认完毕。
- 在这之前，不进入真实加热实现。
- 原因：热敏打印头的驱动与加热时序直接关联功耗、温升和器件安全，不能靠猜测实现。

### Step 08. 测试与联调

- 目标：建立最小可复现测试路径。
- 建议内容：
  - 单元测试优先覆盖协议解析、状态机、缓冲区管理。
  - 联调优先覆盖按键、纸检测、故障输入、走纸流程。

## 4. 对初学 RTOS 的建议

- 先把“分层”和“任务职责”想清楚，再写外设代码，后续会省很多返工。
- 先让任务之间通过清晰的数据结构通信，不要一开始就让所有模块直接互相调用。
- 真正的硬件驱动应当最后接入，因为它最依赖原理图、规格书和示波器验证。
