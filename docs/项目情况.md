# 当前项目情况

更新时间：2026-04-27

本文根据 `项目背景/` 下的真实硬件资料、当前源码和现有文档整理。结论只描述当前已经落地或已经确认的内容，不把“等待实测”写成“已经完成”。

## 1. 总体结论

项目的软件架构和真实硬件方向是匹配的：硬件是一套 `ESP32-S3-WROOM-1-N16R8 + JX-2R-01 热敏机芯 + DRV8833 步进电机驱动 + IP2326 2S 充电管理 + TPS563200 3.3V Buck` 的微型热敏打印机板，当前固件也已经围绕这套硬件建立了 GPIO 映射、安全输出、打印缓存、WiFi API、mock 打印闭环和真实驱动骨架。

但项目还不能宣称“已经真实稳定打印”。当前缺口主要是实物验证和校准：WiFi 联机记录、GPIO 空载波形、VH/STB 安全态、真实走纸方向、真实温度 ADC、电池/充电状态语义、低密度短行真实打印都还需要用户按 `docs/硬件分阶段验收记录.md` 逐步填写。

## 2. 硬件资料匹配判断

### 2.1 已匹配

| 硬件资料结论 | 当前固件状态 | 判断 |
|---|---|---|
| JX-2R-01 规格书给出 384 点/行、8 dots/mm、48mm 有效打印宽度 | `src/config/print_config.h` 定义 384 点、48 字节/行 | 匹配 |
| 规格书把 STB 分成 6 组，每组 64 点 | 代码定义 6 个 STB 组，每组 64 点，并限制同时加热点数 64 | 匹配 |
| 规格书要求不打印时关闭 VH，VH 开关时 STB 保持非工作 | 热敏头驱动和 safe-off 都先关 STB/VH，打开 VH 前检查 STB inactive | 匹配，但需波形验证 |
| 原理图和用户复核确认 `G_VH` 高通低断 | `PIN_G_VH=GPIO8`，安全态低电平关闭 VH | 匹配 |
| 原理图网络到 GPIO 已由用户万用表复核 | `src/bsp/bsp_pins.h` 已写入 `PAPER_N/G_NFAULT/G_NSLEEP/G_KEY/BAT_STAT/EQD/G_VH/LAT/DI/CLK/AIN/BIN/TM1/STB1~6` | 已落地 |
| DRV8833 规格书说明 `nSLEEP` 低电平休眠、`nFAULT` 低有效 | 步进驱动按 `G_NSLEEP` 低休眠、`G_NFAULT` 低故障处理 | 匹配 |
| JX-2R-01 步进距离 0.0625mm，打印点距 0.125mm | 默认参数 `stepsPerPrintLine=2` | 几何关系匹配，方向仍需实测 |
| 原理图 `EQD` 为 220k/100k 电池分压 | 真实传感器骨架按约 3.2 倍换算电池电压 | 初步匹配，需 ADC 校准 |
| 硬件有 `G_KEY` 按键 | 已有按键服务，50ms 轮询、40ms 去抖、短按/长按状态展示 | 初步匹配，有效电平待实测 |
| 用户要求正式控制面用 WiFi | 已有 `/api/v1` HTTP/JSON API 和 `tools/api_client.py` | 软件匹配，实机联网待验证 |

## 3. 项目已有功能

### 3.1 启动与 RTOS 骨架

- `setup()` 最早执行 `Bsp_PreInitSafeOutputs()`，先把 VH、STB、移位信号和 DRV8833 输入收敛到安全态。
- 已创建日志、系统、传感器、参数、打印引擎、命令、协议、监控、WiFi API 等 FreeRTOS 任务入口。
- `src/rtos/` 已集中创建队列、事件组、互斥量和任务注册表。
- `HealthService` 和 `MonitorTask` 已做关键任务心跳检查，关键任务超时会触发危险输出关闭和错误上报。
- 可选 Watchdog 已有 BSP 封装，默认关闭。

### 3.2 板级与硬件抽象

- `src/bsp/bsp_pins.h` 已落地真实 GPIO 映射，不再是全 `PIN_UNASSIGNED`。
- `Bsp_SetAllOutputsSafe()` 统一关闭 VH、STB、打印移位链路和 DRV8833 输出。
- `bsp_gpio` 提供安全 wrapper：无效引脚自动跳过，写入电平会归一化为 LOW/HIGH。
- `bsp_spi`、`bsp_timer`、`bsp_watchdog` 已为真实打印头和系统安全提供底层接口。

### 3.3 WiFi API 正式控制面

启用 `MP_ENABLE_WIFI=1` 后，固件提供 `/api/v1` HTTP/JSON API：

- 基础：`GET /api/v1/info`、`GET /api/v1/status`、`GET /api/v1/health`、`GET /api/v1/logs/recent`。
- 传感器/电池/按键：`GET /api/v1/sensors`、`GET /api/v1/battery`、`GET /api/v1/key`。
- 错误与安全：`GET /api/v1/error`、`POST /api/v1/error/clear`、`POST /api/v1/safe-off`、`POST /api/v1/safe-mode`、`POST /api/v1/reboot`。
- 参数：`GET /api/v1/params`、`PATCH /api/v1/params`、`POST /api/v1/params/save`、`POST /api/v1/params/factory-reset`。
- 打印文件：创建上传、分片写入、完成校验、列表、删除。
- 打印任务：按 `file_id` 启动打印、查询当前任务、取消、走纸。
- 工厂测试：电机小步数测试、打印头 shift/latch 测试、STB 低风险波形测试。

### 3.4 UART 协议兼容路径

- 保留底层 UART 帧解析和响应，用于临时调试/兼容，不再作为正式打印文件传输方案。
- 支持 `PING/GET_INFO/GET_STATUS/GET_ERROR/CLEAR_ERROR/SELF_TEST/REBOOT/GET_PARAM/SAVE_PARAM/FACTORY_RESET/PRINT_START/PRINT_LINE/PRINT_END/PRINT_CANCEL/FEED/SAFE_OFF/SENSOR_TEST/MOTOR_TEST/HEAD_SHIFT_TEST/HEAD_STB_TEST/ENTER_SAFE_MODE`。
- `SAFE_OFF` 在协议层最高优先级处理，SAFE_MODE 下只放行安全白名单命令。

### 3.5 打印数据与打印执行

- 固定行宽：384 点、48 字节/行。
- `PrintSpooler` 使用固定行缓冲池，不在打印接收路径动态分配内存。
- 支持 raw 打印文件上传、CRC32 校验、分片写入和按行入队。
- `PrintEngineTask` 消费行缓冲，执行“安全检查 -> shift/latch/STB/VH -> 走纸 -> 释放行缓冲”的流程。
- 默认走 mock 热敏头和 mock 电机，可以验证软件闭环，不触发真实危险输出。
- 真实热敏头和真实步进电机路径分别受 `MP_ENABLE_HW_THERMAL_HEAD`、`MP_ENABLE_HW_STEPPER` 控制，默认关闭。

### 3.6 传感器、电池、按键与安全

- `SensorService` 汇总纸张、温度、电池电压、充电状态、电机故障，并更新系统事件位。
- 默认 mock 传感器返回有纸、25℃、7400mV、未知充电状态、无电机故障。
- `MP_ENABLE_HW_SENSORS=1` 时启用 ESP32 真实传感器骨架：当前真实读取 `EQD` 电池电压和 `G_NFAULT`，其他项保守占位。
- 低电压保护线当前为 6500mV，低电压会阻止危险打印/电机动作。
- `KeyService` 已接入 `G_KEY`，先按内部上拉、低有效假设处理，只展示状态，不直接触发危险动作。
- `ThermalSafetyService` 已限制缺纸、温度异常、过温、低电压、电机 fault、单组黑点过多和脉冲宽度上限。

### 3.7 参数、NVS、工具与文档

- 参数块包含打印、电机、传感器、通信、日志、安全配置，并带 magic/version/CRC32。
- NVS 读取失败、版本不匹配、长度不匹配或 CRC 错误时回退默认参数。
- WiFi 参数 PATCH 采用白名单和范围校验，不允许未知字段或危险组合。
- 主机工具：
  - `tools/api_client.py`：WiFi API 调试、上传、启动、取消、工厂测试和本地 self-test。
  - `tools/make_test_print.py`：生成 384 点宽 raw 测试打印数据。
  - `tools/send_frame.py`：底层 UART 调试帧工具。
- 文档已包含硬件匹配、GPIO 映射、WiFi API、测试计划、验收记录、待办任务、进度记录等。

## 4. 当前不能算完成的功能

- 还没有真实设备 WiFi API 联机记录。
- 还没有用户填写的安全态、GPIO 空载波形、shift/latch/STB、VH 关闭验证记录。
- 还没有真实打印头加热测试和低密度短行打印结果。
- 还没有真实步进电机正向走纸方向、相序修正和噪声/丢步验证。
- `PAPER_N`、`TM1`、`BAT_STAT` 仍未完成真实语义接入。
- `BAT_STAT` 虽可从 IP2326 规格书确认涓流低电平、退出涓流高电平，但一根状态脚能否满足完整 `CHARGING/FULL/FAULT` 枚举还要继续确认。
- WiFi API 目前适合局域网开发调试，鉴权、配网、mDNS、长期稳定性还不是产品级完成状态。
- 固件侧自动化测试仍不足，`test/` 目录还没有覆盖协议解析、打印缓存、安全白名单、参数边界和热安全计算。

## 5. 下一步建议

1. 按 `docs/硬件分阶段验收记录.md` 从 WiFi API 联机、安全态、sensors、safe-off 开始填写实测结果。
2. 在不打开 VH 的前提下验证 DI/CLK/LAT/STB 波形。
3. 开启 `MP_ENABLE_HW_STEPPER=1` 做低速小步数电机方向测试，确认后再决定是否调整相序。
4. 采集 `PAPER_N` 有纸/无纸电平、`TM1` ADC 原始值、`BAT_STAT` 实际状态。
5. 完成上述低风险验证后，再考虑低密度、短行数、可立即断电的首次真实打印。
