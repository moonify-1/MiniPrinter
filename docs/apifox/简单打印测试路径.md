# Apifox 简单打印测试路径

更新时间：2026-05-21

本文用于在 Apifox 中做最小范围的打印任务测试：先确认设备在线和安全状态，再分别测试“走纸请求”和“一行打印任务”。

所有接口前缀都是：

```text
{{baseUrl}}/api/v1
```

如果串口日志显示：

```text
WiFi STA connected ip=192.168.0.168
```

则 Apifox 环境变量填写：

```text
baseUrl = http://192.168.0.168
```

## 1. 先理解这条测试路径

这条路径故意比 `print_smoke_test.md` 和 `真实打印测试.md` 更短，只做三件事：

| 测试 | 使用接口 | 你能确认什么 |
|---|---|---|
| 走纸请求 | `POST /api/v1/feed?steps=8` | WiFi API 能把走纸命令放进打印控制队列。当前实现仍走 mock 步进电机，所以 API 成功不等于真实纸张会动。 |
| 真实电机小步测试 | `POST /api/v1/factory/motor-test?steps=4` | 真实 DRV8833 步进电机路径是否可用。需要 `MP_ENABLE_HW_STEPPER=1`，否则会返回 `MOTOR_TEST_FAILED`。 |
| 一行打印任务 | 上传 48 字节 raw 文件，再 `POST /api/v1/print/jobs` | 打印文件上传、CRC 校验、打印任务入队、1 行打印流程是否闭环。真实纸面输出还要求真实热敏头和真实步进电机宏都已启用。 |
| 一键真实纸面测试 | `POST /api/v1/factory/real-print-test?lines=16` | 固件内部生成低密度点迹并启动真实打印任务，用来快速看到纸面实际效果。 |

判断时一定分清三层：

1. API 返回成功：说明请求被固件接受。
2. 任务状态成功：说明 RTOS 打印任务处理了请求。
3. 纸真的动或纸面真的有点：说明真实硬件路径、供电、传感器和机芯都在工作。

## 2. 测试文件

| 项目 | 值 |
|---|---|
| raw 文件 | `docs/apifox/payloads/print_simple_1line.bin` |
| 图案 | 一行低密度竖向标记点，用来观察是否有一行点迹 |
| 行宽 | `48 bytes/line` |
| 行数 | `1` |
| 总大小 | `48 bytes` |
| CRC32/IEEE 十六进制 | `0x5F00F6CD` |
| CRC32/IEEE 十进制 | `1593898701` |
| 分片序号 | `0` |
| 每个 STB 组黑点数 | `8/8/8/8/8/8` |

这个文件非常短，适合第一次验证“打印一行”的 API 链路。因为只有 1 行，如果真实打印成功，纸面上通常只会看到很细的一行稀疏点迹，不会像完整图片那样明显。

## 3. 一键真实打印脚本

如果你现在要看的不是 API 闭环，而是实际纸面效果，优先用仓库里的真实打印脚本：

```powershell
.\tools\real_simple_print_test.ps1
```

这个脚本会先检查 `GET /api/v1/info`，只有 `features.hw_stepper=true` 且 `features.hw_thermal_head=true` 时才会继续。它不会使用当前仍为 mock 路径的 `/api/v1/feed`，而是先调用 `POST /api/v1/factory/motor-test?steps=4` 验证真实电机路径，再调用 `POST /api/v1/factory/real-print-test?lines=16&density=25&heat=25`，由固件内部生成和 `print_simple_1line.bin` 相同的低密度点迹并启动真实打印任务。这样纸面效果比单独打印 1 行更容易看到。

注意：你这次运行卡在 `PUT /api/v1/print/files/{file_id}/chunks/0`，说明当前固件的 raw 二进制上传路径还会导致超时或重启。新的脚本路径不再执行这个 `PUT` 上传步骤；如果运行后返回 `NOT_FOUND`，说明设备还没有烧录包含 `/api/v1/factory/real-print-test` 的新固件。

如果电机测试失败、设备处于 `SAFE_MODE`、或者固件不是全真实硬件路径，脚本会中止，并在退出前尽量调用 `safe-off` 关闭危险输出。

常用参数：

```powershell
.\tools\real_simple_print_test.ps1 -RepeatLines 24 -Density 25 -Heat 25
```

脚本默认设备地址已经固定为 `http://192.168.0.168`。其中 `RepeatLines` 只是把同一行测试点迹多打印几行，便于观察；`Density` 和 `Heat` 仍保持低能量测试思路，第一次真实打印不建议直接调高。

## 4. Apifox 通用设置

每个请求建议带：

| Header | 值 |
|---|---|
| `Accept` | `application/json` |

上传二进制分片时还必须带：

| Header | 值 |
|---|---|
| `Content-Type` | `application/octet-stream` |

如果 Body 写着 `none`，就不要填 JSON、Text、form-data 或 Binary。

## 5. Step 00：查询功能开关

### 作用

确认设备在线，并确认当前固件是否启用了真实硬件路径。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `GET` |
| URL | `{{baseUrl}}/api/v1/info` |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "features": {
    "hw_thermal_head": false,
    "hw_stepper": false
  }
}
```

重点看：

| 字段 | 含义 |
|---|---|
| `features.hw_stepper` | `true` 才可能用 `factory/motor-test` 看到真实电机动作。 |
| `features.hw_thermal_head` | `true` 才可能真实加热点阵。 |

如果两个字段都是 `false`，后面的打印任务仍可测试 API 闭环，但不会真实出纸或加热。

## 6. Step 01：安全关闭危险输出

### 作用

测试前先关闭 VH、STB、电机等危险输出。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/safe-off` |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK"
}
```

## 7. Step 02：查询整机状态

### 作用

确认设备没有处于 `SAFE_MODE`，并检查纸张、温度、电池、电机故障摘要。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `GET` |
| URL | `{{baseUrl}}/api/v1/status` |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "system_state": "READY"
}
```

`system_state` 不能是 `SAFE_MODE`。如果处于 `SAFE_MODE`，走纸、工厂测试和启动打印都会被拒绝。

## 8. Step 03：测试走纸请求

### 作用

测试 `/api/v1/feed` 这个业务走纸 API 是否能入队。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/feed?steps=8` |
| Query 参数 `steps` | `8` |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK",
  "steps": 8
}
```

注意：当前源码中 `FEED` 控制消息仍调用 mock stepper，所以这一步成功只代表“走纸请求已被打印任务处理”，不代表真实纸张一定会动。

## 9. Step 03B：真实电机小步测试

### 作用

如果你现在想看真实纸张是否会动，用这个工厂测试接口，而不是只看 `/feed`。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/factory/motor-test?steps=4` |
| Query 参数 `steps` | `4` |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK",
  "steps": 4
}
```

如果返回：

```json
{
  "ok": false,
  "code": "MOTOR_TEST_FAILED"
}
```

常见含义是：

| 可能原因 | 下一步 |
|---|---|
| 固件没有启用 `MP_ENABLE_HW_STEPPER=1` | 重新编译烧录真实电机测试固件。 |
| `nFAULT` 报故障 | 先检查 DRV8833、电机排线、供电和引脚映射。 |
| 电池电压低或传感器异常 | 先查 `GET /api/v1/sensors` 和 `GET /api/v1/battery`。 |

## 10. Step 04：创建一行打印文件上传会话

### 作用

告诉固件：我要上传一个 48 字节，也就是 1 行的 raw 打印文件。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/print/files` |
| Query 参数 `size` | `48` |
| Query 参数 `crc32` | `0x5F00F6CD` |
| Body | `none` |
| Header | `Accept: application/json` |

完整 URL 也可以直接写成：

```text
{{baseUrl}}/api/v1/print/files?size=48&crc32=0x5F00F6CD
```

### 成功响应重点

```json
{
  "ok": true,
  "file": {
    "file_id": 1,
    "state": "RECEIVING",
    "total_bytes": 48,
    "received_bytes": 0,
    "line_count": 1,
    "expected_crc32": 1593898701
  }
}
```

把 `file.file_id` 记下来，后续 `{file_id}` 都填这个值。

## 11. Step 05：上传一行二进制数据

### 作用

把 `print_simple_1line.bin` 的真实二进制内容传给固件。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `PUT` |
| URL | `{{baseUrl}}/api/v1/print/files/{file_id}/chunks/{index}` |
| Path 参数 `file_id` | Step 04 返回的 `file.file_id`，例如 `1` |
| Path 参数 `index` | `0` |
| Body | 选择 `Binary` |
| Binary 文件 | `docs/apifox/payloads/print_simple_1line.bin` |
| Header `Accept` | `application/json` |
| Header `Content-Type` | `application/octet-stream` |

如果 `file_id = 1`，完整 URL 是：

```text
{{baseUrl}}/api/v1/print/files/1/chunks/0
```

Body 只能选择 `Binary`，不要选择 JSON、Text 或 form-data。

### 成功响应重点

```json
{
  "ok": true,
  "file": {
    "total_bytes": 48,
    "received_bytes": 48
  }
}
```

如果 `received_bytes` 不是 `48`，先不要执行下一步，重新检查 Body 是否真的选择了 `.bin` 文件。

## 12. Step 06：完成上传并校验 CRC

### 作用

让固件检查字节数和 CRC32。只有 `COMPLETE` 文件才能启动打印任务。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/print/files/{file_id}/complete` |
| Path 参数 `file_id` | Step 04 返回的 `file.file_id` |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "file": {
    "state": "COMPLETE",
    "line_count": 1,
    "actual_crc32": 1593898701
  }
}
```

必须看到：

```text
state        = COMPLETE
actual_crc32 = 1593898701
```

## 13. Step 07：启动一行打印任务

| URL | `{{baseUrl}}/api/v1/print/jobs?file_id={file_id}&copies=1&density=25&heat=25` |
| Query 参数 `file_id` | Step 04 返回的 `file.file_id` |
| Query 参数 `copies` | `1` |
| Query 参数 `density` | `25` |
| Query 参数 `heat` | `25` |
| Body | `none` |
| Header | `Accept: application/json` |

如果 `file_id = 1`，完整 URL 是：

```text
{{baseUrl}}/api/v1/print/jobs?file_id=1&copies=1&density=25&heat=25
```

### 成功响应重点

```json
{
  "ok": true,
  "job_id": 1,
  "file_id": 1,
  "line_total": 1,
  "copies": 1
}
```

如果当前固件是 mock 硬件路径，接口可能成功，任务也可能完成，但纸面不会有真实输出。

## 14. Step 08：查询当前打印任务

### 作用

查看任务是否已经处理完。1 行任务很短，可能很快从 `PRINTING` 变成 `DONE`。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `GET` |
| URL | `{{baseUrl}}/api/v1/print/jobs/current` |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "state": "DONE",
  "line_done": 1,
  "error": 0
}
```

`state` 也可能短暂看到 `QUEUED` 或 `PRINTING`。如果看到 `FAILED`，先记录 `error`，再执行 `safe-off`。

## 17. 结果判断表

| 现象 | 更可能说明什么 |
|---|---|
| `/feed` 返回 OK，但纸不动 | 当前 `FEED` 仍是 mock 步进电机路径，不能证明真实电机失败。 |
| `/factory/motor-test` 返回 OK，纸轻微动 | 真实步进电机路径、方向和 release/sleep 初步可用。 |
| `/factory/motor-test` 返回 `MOTOR_TEST_FAILED` | 真实电机宏未开、nFAULT 异常、低电压或硬件连接异常。 |
| 一行打印任务返回 OK 且 `line_total=1` | 上传、CRC、打印任务入队链路通过。 |
| 一行任务 `DONE`，但纸面空白 | 可能仍是 mock 硬件路径，或真实热敏头/VH/纸张热敏面未满足条件。 |
| 纸不动但打印头发热 | 立即 `safe-off` 或断电，再检查电机、FPC、VH/STB 和固件宏，不要继续加热测试。 |

## 18. 新手理解：为什么一行是 48 字节

当前 raw 打印格式固定为 `384 dots` 宽。1 个字节有 8 个 bit，所以一行需要：

```text
384 / 8 = 48 bytes
```

固件上传服务要求文件大小必须是 `48` 的整数倍。本文的一行测试文件正好是 `48 bytes`，因此 `line_count = 1`。
