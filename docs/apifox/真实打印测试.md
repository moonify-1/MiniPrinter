# Apifox 真实打印测试

更新时间：2026-04-27

本文用于在 Apifox 中手工调用 MiniPrinterRTOS WiFi API，进行一次真实打印测试。  
按 `00 -> 08` 顺序执行，不要跳步。所有接口前缀都是：

```text
{{baseUrl}}/api/v1
```

如果串口日志显示：

```text
WiFi STA connected ip=192.168.0.168
```

则 Apifox 环境变量填写：

```text
baseUrl = http://192.168.0.168
```

## 1. 真实打印前置条件

这份文档不是低密度冒烟测试，执行前请确认：

1. 已重新烧录包含 Step 55 二进制上传修复的固件。
2. 已完成 `safe-off`、`head-shift-test`、`head-stb-test`、`motor-test` 等空载测试。
3. 真实打印头、步进电机、纸张、电池、电机故障、温度保护都已确认可用。
4. 如果当前固件仍是 mock 硬件路径，API 会成功，但不会真实加热或走纸。

真实打印过程中如果出现异味、过热、走纸卡住、异常噪声、纸张不动但仍在加热，立即调用：

```text
POST {{baseUrl}}/api/v1/safe-off
```

## 2. 测试文件

| 项目 | 值 |
|---|---|
| 完整 raw 文件 | `docs/apifox/payloads/print_real_moderate_48lines.bin` |
| 图案 | 中等长度真实打印图案，包含边框、居中文字、刻度和走纸方向观察线 |
| 行宽 | `48 bytes/line` |
| 行数 | `48` |
| 总大小 | `2304 bytes` |
| CRC32/IEEE 十六进制 | `0x282EB33A` |
| CRC32/IEEE 十进制 | `674149178` |
| 分片大小 | `512 bytes`，最后一片 `256 bytes` |
| 分片数量 | `5` |
| 最大单 STB 组黑点数 | `33`，低于当前安全上限 `64` |

当前上传服务最大支持 `3072 bytes / 64 lines`。这个文件选 `48 lines`，目的是比 4 行冒烟测试更接近真实输出，又不直接打满最大长度。

## 3. Apifox 通用设置

每个请求都建议带：

| Header | 值 |
|---|---|
| `Accept` | `application/json` |

上传二进制分片的 Step 03 还必须额外带：

| Header | 值 |
|---|---|
| `Content-Type` | `application/octet-stream` |

如果 Apifox 页面里有 `Path` 参数和 `Query` 参数，按本文对应步骤填写。  
如果 Body 写着 `none`，就不要填 JSON、Text、form-data 或 Binary。

## 4. Step 00：安全关闭危险输出

### 作用

测试前先关闭 VH、STB、电机等危险输出。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/safe-off` |
| Path 参数 | 无 |
| Query 参数 | 无 |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK"
}
```

## 5. Step 01：确认设备和安全状态

### 作用

确认设备在线，并确认当前没有处于 `SAFE_MODE`。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `GET` |
| URL | `{{baseUrl}}/api/v1/status` |
| Path 参数 | 无 |
| Query 参数 | 无 |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "system_state": "READY"
}
```

`system_state` 不能是 `SAFE_MODE`。如果是 `SAFE_MODE`，启动打印会被拒绝。

建议同时看响应里的传感器摘要：

| 字段 | 期望 |
|---|---|
| `sensors.paper` | `true`，有纸 |
| `sensors.motor_fault` | `false` |
| `sensors.validity` | 可用状态 |
| `sensors.head_temp_c` | 不能接近过温阈值 |

如果你想单独查传感器，可额外调用：

```text
GET {{baseUrl}}/api/v1/sensors
```

## 6. Step 02：创建真实打印文件上传会话

### 作用

告诉固件：我要上传一个 `2304 bytes` 的 raw 打印文件，它的 CRC32 是 `0x282EB33A`。固件会分配一个 `file_id`。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/print/files` |
| Path 参数 | 无 |
| Query 参数 `size` | `2304` |
| Query 参数 `crc32` | `0x282EB33A` |
| Body | `none` |
| Header | `Accept: application/json` |

完整 URL 也可以直接写成：

```text
{{baseUrl}}/api/v1/print/files?size=2304&crc32=0x282EB33A
```

注意参数名是 `crc32`，不是 `srs32`。

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK",
  "file": {
    "file_id": 1,
    "state": "RECEIVING",
    "total_bytes": 2304,
    "received_bytes": 0,
    "line_count": 48,
    "expected_crc32": 674149178
  }
}
```

把返回里的 `file.file_id` 记下来。后续 `{file_id}` 都填这个值。

例如返回 `file_id = 1`，后面就用：

```text
file_id = 1
```

## 7. Step 03：上传 5 个二进制分片

### 作用

把真实打印文件按固件定义的 `512 bytes/chunk` 上传。  
每次请求 Body 都选择 `Binary`，并选择对应的 chunk 文件。

### 分片清单

| PUT 次数 | Path 参数 `index` | Binary 文件 | 文件大小 | 成功后 `received_bytes` |
|---:|---:|---|---:|---:|
| 1 | `0` | `docs/apifox/payloads/print_real_moderate_48lines_chunk_0.bin` | `512` | `512` |
| 2 | `1` | `docs/apifox/payloads/print_real_moderate_48lines_chunk_1.bin` | `512` | `1024` |
| 3 | `2` | `docs/apifox/payloads/print_real_moderate_48lines_chunk_2.bin` | `512` | `1536` |
| 4 | `3` | `docs/apifox/payloads/print_real_moderate_48lines_chunk_3.bin` | `512` | `2048` |
| 5 | `4` | `docs/apifox/payloads/print_real_moderate_48lines_chunk_4.bin` | `256` | `2304` |

### 每次 PUT 的 Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `PUT` |
| URL | `{{baseUrl}}/api/v1/print/files/{file_id}/chunks/{index}` |
| Path 参数 `file_id` | Step 02 返回的 `file.file_id`，例如 `1` |
| Path 参数 `index` | 按上表填写 `0`、`1`、`2`、`3`、`4` |
| Query 参数 | 无 |
| Body | 选择 `Binary` |
| Binary 文件 | 按上表选择对应 chunk 文件 |
| Header `Accept` | `application/json` |
| Header `Content-Type` | `application/octet-stream` |

如果 Step 02 返回 `file_id = 1`，5 次 PUT 的完整 URL 是：

```text
PUT {{baseUrl}}/api/v1/print/files/1/chunks/0
PUT {{baseUrl}}/api/v1/print/files/1/chunks/1
PUT {{baseUrl}}/api/v1/print/files/1/chunks/2
PUT {{baseUrl}}/api/v1/print/files/1/chunks/3
PUT {{baseUrl}}/api/v1/print/files/1/chunks/4
```

Body 只能选择 `Binary`，不要选择 `JSON`、`Text`、`form-data`，也不要在 Body 里手打数字或文件名。

### 第 5 片成功响应重点

```json
{
  "ok": true,
  "code": "OK",
  "file": {
    "file_id": 1,
    "state": "RECEIVING",
    "total_bytes": 2304,
    "received_bytes": 2304
  }
}
```

第 5 片完成后必须看到：

```text
received_bytes = 2304
total_bytes    = 2304
```

如果中途某一次 `received_bytes` 没有按表增加，不要执行 Step 04。先检查当前 PUT 的 `index` 和 Binary 文件是否对应。

## 8. Step 04：完成上传并校验 CRC

### 作用

让固件检查上传是否完整，并重新计算 CRC32。校验通过后，文件状态才会变成 `COMPLETE`，后面才能启动打印。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/print/files/{file_id}/complete` |
| Path 参数 `file_id` | Step 02 返回的 `file.file_id`，例如 `1` |
| Query 参数 | 无 |
| Body | `none` |
| Header | `Accept: application/json` |

如果 Step 02 返回 `file_id = 1`，完整 URL 就是：

```text
{{baseUrl}}/api/v1/print/files/1/complete
```

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK",
  "file": {
    "file_id": 1,
    "state": "COMPLETE",
    "total_bytes": 2304,
    "received_bytes": 2304,
    "line_count": 48,
    "expected_crc32": 674149178,
    "actual_crc32": 674149178
  }
}
```

必须看到：

```text
state        = COMPLETE
actual_crc32 = 674149178
```

### 常见失败

| 现象 | 处理 |
|---|---|
| `received_bytes < 2304` | Step 03 分片没有传全，回到缺失的 `index` 重新上传 |
| `received_bytes = 2304` 但 CRC 失败 | Step 02 的 `crc32` 填错，或 Step 03 某个 chunk 文件选错 |
| `PRINT_FILE_COMPLETE_FAILED` | 先执行 Step 08 的文件列表查询，看每个字段是否符合本文 |

## 9. Step 05：启动真实打印任务

### 作用

根据已经 `COMPLETE` 的文件创建打印任务。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/v1/print/jobs` |
| Path 参数 | 无 |
| Query 参数 `file_id` | Step 02 返回的 `file.file_id`，例如 `1` |
| Query 参数 `copies` | `1` |
| Query 参数 `density` | `35` |
| Query 参数 `heat` | `35` |
| Body | `none` |
| Header | `Accept: application/json` |

如果 `file_id = 1`，完整 URL 可以写成：

```text
{{baseUrl}}/api/v1/print/jobs?file_id=1&copies=1&density=35&heat=35
```

当前 v1 只支持：

```text
copies = 1
density = 0..100
heat    = 0..100
```

说明：当前固件里 `density` 和 `heat` 主要做 API 范围校验；真实热脉冲仍由 `ParamBlock` 里的安全参数、温度、电压和每组黑点数计算。

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK",
  "job_id": 1,
  "file_id": 1,
  "line_total": 48,
  "copies": 1
}
```

如果这一步返回成功，立即观察真实纸面：

| 观察项 | 期望 |
|---|---|
| 是否出纸 | 应该稳定走纸 |
| 是否有图案 | 应能看到边框、文字和刻度类图案 |
| 方向 | 图案不应左右错乱到完全不可辨认 |
| 发热 | 不应持续过热或有异常气味 |
| 电机 | 不应卡住、尖叫或触发故障 |

## 10. Step 06：查询当前打印任务

### 作用

查看当前打印任务状态。48 行文件仍然较短，可能很快完成，所以状态可能已经是 `DONE` 或 `NONE`。

### Apifox 填写

| 项目 | 填写 |
|---|---|
| Method | `GET` |
| URL | `{{baseUrl}}/api/v1/print/jobs/current` |
| Path 参数 | 无 |
| Query 参数 | 无 |
| Body | `none` |
| Header | `Accept: application/json` |

### 成功响应重点

```json
{
  "ok": true,
  "code": "OK",
  "job_id": 1,
  "file_id": 1,
  "state": "DONE",
  "line_done": 48,
  "error": 0
}
```

常见状态：

| state | 含义 |
|---|---|
| `QUEUED` | 已排队 |
| `PRINTING` | 正在打印 |
| `DONE` | 已完成 |
| `FAILED` | 打印失败 |
| `CANCELLED` | 已取消 |
| `NONE` | 当前没有活动任务 |

如果看到 `FAILED`，先记录响应里的 `error`，再执行 `safe-off`。

## 13. 一次成功测试的最短参数清单

假设设备 IP 是 `192.168.0.168`，Step 02 返回 `file_id=1`，则完整流程如下：

```text
00 POST http://192.168.0.168/api/v1/safe-off
01 GET  http://192.168.0.168/api/v1/status
02 POST http://192.168.0.168/api/v1/print/files?size=2304&crc32=0x282EB33A

03 PUT  http://192.168.0.168/api/v1/print/files/1/chunks/0
   Body: Binary -> docs/apifox/payloads/print_real_moderate_48lines_chunk_0.bin
03 PUT  http://192.168.0.168/api/v1/print/files/1/chunks/1
   Body: Binary -> docs/apifox/payloads/print_real_moderate_48lines_chunk_1.bin
03 PUT  http://192.168.0.168/api/v1/print/files/1/chunks/2
   Body: Binary -> docs/apifox/payloads/print_real_moderate_48lines_chunk_2.bin
03 PUT  http://192.168.0.168/api/v1/print/files/1/chunks/3
   Body: Binary -> docs/apifox/payloads/print_real_moderate_48lines_chunk_3.bin
03 PUT  http://192.168.0.168/api/v1/print/files/1/chunks/4
   Body: Binary -> docs/apifox/payloads/print_real_moderate_48lines_chunk_4.bin

04 POST http://192.168.0.168/api/v1/print/files/1/complete
05 POST http://192.168.0.168/api/v1/print/jobs?file_id=1&copies=1&density=35&heat=35
06 GET  http://192.168.0.168/api/v1/print/jobs/current
07 GET  http://192.168.0.168/api/v1/print/files
08 POST http://192.168.0.168/api/v1/safe-off
08 DELETE http://192.168.0.168/api/v1/print/files/1
```

所有 PUT 请求都要带：

```text
Content-Type: application/octet-stream
Accept: application/json
```

## 14. 失败时先看这些字段

| 失败位置 | 先看字段 | 正确值 |
|---|---|---|
| Step 02 | `total_bytes` | `2304` |
| Step 02 | `line_count` | `48` |
| Step 03 index 0 | `received_bytes` | `512` |
| Step 03 index 1 | `received_bytes` | `1024` |
| Step 03 index 2 | `received_bytes` | `1536` |
| Step 03 index 3 | `received_bytes` | `2048` |
| Step 03 index 4 | `received_bytes` | `2304` |
| Step 04 | `state` | `COMPLETE` |
| Step 04 | `actual_crc32` | `674149178` |
| Step 05 | `line_total` | `48` |

如果真实纸面完全空白，但 API 显示 `DONE`，优先检查：

1. 当前固件是否真的启用了真实热敏头和真实步进电机宏。
2. VH 电源是否存在。
3. `head-shift-test` 和 `head-stb-test` 是否已经用示波器确认。
4. 纸张热敏面方向是否正确。
5. `GET /api/v1/sensors` 是否显示缺纸、低电压、温度无效或电机故障。
