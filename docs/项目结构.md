# 项目结构

## 1. 当前情况分析

- 本项目是 PlatformIO + Arduino framework 的 ESP32-S3 固件工程，目标板为 `esp32-s3-devkitc-1`。
- 项目已经开始按 `docs/architecture_notes.md` 定义的分层落地，不再是只有 `src/main.cpp` 的空骨架。
- 当前代码使用 FreeRTOS 对象和任务组织业务：`src/rtos/` 创建队列、事件组、互斥量；`src/tasks/` 创建各个任务入口。
- 当前开发配置 `src/config/project_config.h` 已默认开启 WiFi API、真实步进电机和真实热敏头宏；传感器、Watchdog、BLE 仍保持默认关闭。`/api/v1/factory/motor-test` 和打印任务会走真实硬件路径，`/api/v1/feed` 仍是 mock 走纸路径。
- `docs/pin_map.md` 和 `src/bsp/bsp_pins.h` 已写入用户复核过的 GPIO 映射，但真实输出行为仍需要空载波形和安全态验证。
- WiFi API 已成为正式产品控制面方向：启用 `MP_ENABLE_WIFI=1` 后提供 `/api/v1` HTTP/JSON 接口；UART 协议只保留为底层临时调试/兼容路径，不再作为正式打印文件传输或启动打印方案。
- 启动流程安全优先：`setup()` 里最早执行 `Bsp_PreInitSafeOutputs()`，之后才初始化串口、RTOS 对象、服务和任务。
- 打印链路已经具备雏形：协议命令进入队列，打印行进入 Spooler，PrintEngineTask 消费行数据，打印安全检查负责限温、限点数、限脉宽。
- 协议层已有基础 UART 帧解析和响应：支持 `PING`、状态查询、错误查询、参数保存、打印、走纸和工厂测试命令。
- 安全链路已经有基础闭环：ErrorService 上报错误，SystemTask 根据严重级别进入 `ERROR` 或 `SAFE_MODE`，`SAFE_OFF` 可最高优先级关闭危险输出。
- 主机侧工具已经存在：`tools/send_frame.py` 用于构造/发送底层调试协议帧，`tools/make_test_print.py` 用于生成 384 点宽测试打印数据，`tools/real_simple_print_test.py` 和 `tools/real_simple_print_test.ps1` 用于一键执行真实简单打印验收；一键脚本当前调用 `/api/v1/factory/real-print-test`，由固件内部生成低密度样张，避开主机二进制上传路径。

## 2. 当前验证结果

- `python -m platformio run` 编译通过；当前默认配置已包含 WiFi、真实热敏头和真实步进电机宏，当前编译结果约为 RAM 21.3%、Flash 24.2%。
- 历史验证中，`PLATFORMIO_BUILD_FLAGS="-DMP_ENABLE_WIFI=1"` 和 `PLATFORMIO_BUILD_FLAGS="-DMP_ENABLE_HW_THERMAL_HEAD=1 -DMP_ENABLE_HW_STEPPER=1"` 的临时覆盖编译均通过。
- `python tools/send_frame.py --help` 正常运行。
- `python tools/make_test_print.py --help` 正常运行。
- `python tools/real_simple_print_test.py --dry-run` 和 `powershell -ExecutionPolicy Bypass -File tools\real_simple_print_test.ps1 -DryRun` 均通过。

## 3. 当前主要风险

- 已看到真实设备 WiFi API 联机：`/api/v1/info`、`/api/v1/safe-off`、`/api/v1/status` 和 `/api/v1/factory/motor-test?steps=4` 均返回成功；真实纸面输出仍需在烧录包含 `/api/v1/factory/real-print-test` 的新固件后复测。
- 旧的 `PUT /api/v1/print/files/{file_id}/chunks/{index}` raw 二进制上传路径在实机上出现过超时/设备重启现象，后续如果还要支持通用图片上传，需要单独排查 WebServer raw body 处理。
- WiFi API 已覆盖基础状态、打印文件上传、启动打印、取消、走纸、工厂测试和参数接口，当前主要缺少真实设备联机记录和固件侧自动化测试。
- 参数 WiFi PATCH 已覆盖安全基础字段，后续还需要继续补齐电机/通信等白名单字段和边界验证。
- 传感器目录已有受宏保护的 ESP32 真实传感器骨架；`PAPER_N`、`TM1`、`BAT_STAT` 仍按保守策略处理，真实电平和 ADC 校准需要用户实测。
- `test/` 目前只有 README，协议解析、缓冲区边界、状态机和安全路径还缺少自动化测试。
- `ProtocolTask` 当前使用 Serial 简单轮询，后续如果吞吐压力变大，可以再评估 ring buffer、DMA 或中断驱动接收。

## 5. 源码分层说明

- `src/config/`：编译期开关、打印参数、安全限制、默认参数。
- `src/common/`：错误码、公共类型、固定内存池等硬件无关基础能力。
- `src/bsp/`：板级支持包，负责引脚、安全输出、SPI、微秒延时、Watchdog 等板级资源。
- `src/drivers/`：具体器件驱动接口和实现，例如热敏打印头、DRV8833 步进电机、NVS 存储、传感器 mock 和受宏保护的 ESP32 真实传感器骨架。
- `src/services/`：可复用业务服务，例如日志、错误、健康检查、参数、传感器、按键、协议、打印、打印文件上传、安全计算和 WiFi API 收发。
- `src/protocol/`：命令号、帧格式、CRC 和逐字节解析器。
- `src/app/`：设备级业务状态，例如系统状态机、打印作业状态、工厂测试动作。
- `src/tasks/`：FreeRTOS 任务入口，负责调度服务和应用层逻辑；`task_wifi_api` 负责在启用 WiFi 时轮询 HTTP 客户端。
- `src/rtos/`：FreeRTOS 队列、事件组、互斥量和任务注册表。
- `tools/`：PC 侧辅助脚本，用于 WiFi API 验收、真实简单打印验收、串口底层调试和测试打印数据生成。
- `docs/`：正式设计、验收、WiFi API、Apifox/OpenAPI 导入数据源、Apifox 简单打印测试、打印冒烟测试集合、真实打印测试指导、项目启动说明和进度文档。
- `docx/`：保留的文档快照和修改记录目录。
- `项目背景/`：实际硬件资料目录，保存原理图、PCB、BOM、GPIO 映射、外设数据手册等；其中 `ESP32GPIO映射图.md` 是对原 GPIO 图片的 Markdown 速查整理。

## 6. 当前建议下一步

- 按 `docs/硬件分阶段验收记录.md` 先完成 WiFi API 联机、安全态、sensors、battery 和 safe-off 实测记录。
- 在不开 VH 的前提下验证 GPIO、shift/latch、STB 测试命令的波形。
- 采集 `PAPER_N` 有纸/无纸电平、`TM1` ADC 原始值、`BAT_STAT` 实际状态和电机正向走纸方向。
- 最后才逐步开启真实打印头和步进电机宏，并继续保持低能量、短脉宽、低速验证。
