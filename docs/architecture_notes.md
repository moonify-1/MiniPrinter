# MiniPrinter 当前软件分层说明

## 1. 说明

当前仓库物理上仍是 PlatformIO 默认骨架，逻辑分层还没有正式落地。  
本文先定义“推荐的软件分层”，后续代码目录按这份说明逐步展开。

## 2. 分层目标

### `config`

- 作用：集中放编译期开关、默认参数、板级常量、任务栈大小、队列长度等配置。
- 特点：尽量少放逻辑，多放“可查、可改、可比较”的常量。

### `common`

- 作用：放全项目共享的基础类型、通用宏、错误码、日志接口、轻量工具函数。
- 约束：`common` 不直接依赖具体硬件，不要把 GPIO 或芯片寄存器操作放进来。

### `bsp`

- 全称：Board Support Package，板级支持包。
- 作用：描述“这块板子上有什么资源，以及资源接到了哪里”。
- 典型内容：GPIO 映射、引脚初始化、板级电源使能脚、按键脚、故障脚。
- 说明：它关心“板子连线”，但不负责复杂业务。

### `drivers`

- 作用：封装具体器件或具体外设的访问接口。
- 典型内容：
  - 步进电机驱动芯片接口。
  - 热敏打印头移位时序接口。
  - 电池状态输入读取接口。
- 约束：`drivers` 只提供能力，不直接决定“什么时候打印、什么时候走纸”。

### `services`

- 作用：承接可复用的业务能力，把多个驱动组合成一个“服务”。
- 典型内容：
  - 走纸服务。
  - 打印缓存服务。
  - 设备状态汇总服务。
- 特点：它知道“怎么完成一件事”，但不一定知道“为什么此刻要做”。

### `protocol`

- 作用：处理外部通信协议。
- 典型内容：
  - 命令字定义。
  - 数据帧结构。
  - 接收解析状态机。
  - 响应打包。
- 约束：协议层尽量不要直接操作硬件，而是把结果交给 `app` 或 `services`。

### `app`

- 作用：放设备级业务流程，是“产品行为”的核心层。
- 典型内容：
  - 打印任务编排。
  - 空闲 / 走纸 / 打印 / 故障 等状态流转。
  - 用户命令到具体服务的调度。

### `tasks`

- 作用：放 RTOS 任务入口与任务间协作代码。
- 典型内容：
  - 通信任务。
  - 控制任务。
  - 状态监控任务。
- 说明：`tasks` 更关心“哪个任务在运行、用什么节奏运行”，而不是业务细节本身。

### `rtos`

- 作用：放 RTOS 适配层与系统级封装。
- 典型内容：
  - 任务创建封装。
  - 队列、信号量、事件组的统一包装。
  - Tick、延时、临界区等系统接口。
- 价值：把上层代码从具体 RTOS API 中稍微隔离出来，后面更容易维护。

### `test`

- 作用：放单元测试、集成测试、假实现、测试数据。
- 建议：优先测试协议解析、状态机、缓冲区边界，而不是一上来测试真实引脚。

## 3. 推荐依赖方向

建议按下面方向组织依赖，尽量避免上层和底层互相穿透：

`config/common -> bsp/drivers -> services/protocol -> app -> tasks -> rtos 启动入口`

补充说明：

- `common` 和 `config` 是公共底座。
- `app` 不应直接操作底层 GPIO。
- `tasks` 负责调度，尽量少放业务判断。
- `drivers` 不应反向依赖 `app`。

## 4. 推荐目录映射

下面是后续比较容易维护的一种目录形态：

```text
src/
  app/
  bsp/
  common/
  config/
  drivers/
  protocol/
  rtos/
  services/
  tasks/
include/
test/
```

## 5. 当前状态

- 当前物理代码仍只有 `src/main.cpp` 模板文件。
- 上述分层目前属于“已定义、未落地”状态。
- 下一步更合适的动作是先创建最小目录骨架和 RTOS 入口，再逐层填充。
