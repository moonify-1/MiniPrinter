# 硬件匹配分析

更新时间：2026-04-26

本文基于 `项目背景/` 目录下的原理图、BOM、ESP32-S3 DevKitC-1 GPIO 映射图和关键器件规格书，对照当前固件实现，判断硬件是否匹配、哪些功能还没有匹配到代码。

## 1. 结论

当前固件的总体方向与硬件是匹配的：硬件确实是一套基于 ESP32-S3、JX-2R-01 热敏机芯、DRV8833 步进电机驱动、IP2326 2S 充电管理、TPS563200 3.3V Buck 的小型热敏打印机板。

但当前代码仍处于“软件闭环 + 安全骨架”阶段，尚未真正匹配完整硬件。最关键的差距有三类：

- `src/bsp/bsp_pins.h` 仍然全部是 `PIN_UNASSIGNED`，还没有把原理图上的网络映射到 ESP32 GPIO。
- 硬件有真实纸检测、热敏电阻、电池电压分压、充电状态、按键输入，但当前 `SensorDriver` 仍是 mock，且缺少 `EQD` 电池电压 ADC 网络。
- 打印头和电机的安全框架已经有了，但热脉冲模型、电机步进节奏、真实 ADC 换算、真实串口/GPIO/波形验证还没有按规格书闭环。

## 3. 原理图上的主要硬件模块

### 3.1 打印机芯 FPC

原理图中的 FPC 基本符合 JX-2R-01 规格书表 15 的接口定义：

| 机芯信号 | 规格书接口 | 原理图连接 | 当前代码状态 |
| --- | --- | --- | --- |
| `VSEN` | Pin 2 | `BUCK_OUT` 经过 `R21=180R` | 未单独建模 |
| `PHE` | Pin 3 | 进入 LM393 缺纸检测电路 | 只在硬件中存在，代码未直接控制 |
| `VH` | Pin 6/7/25/26 | 由 `G_VH` 控制的高边 P-MOS 输出 `VH_OUT` | 有 VH 控制骨架，但 GPIO 未分配 |
| `DI` | Pin 8 | 经过 22R 接 MCU | 有 SPI shift 骨架，但 GPIO 未分配 |
| `CLK` | Pin 9 | 经过 22R 接 MCU | 有 SPI clock 骨架，但 GPIO 未分配 |
| `GND` | Pin 10/11/21/22 | 接地 | 匹配 |
| `STB1~STB6` | Pin 20/19/18/14/13/12 | 经过 22R 接 MCU | 有 6 组 STB 控制骨架，但 GPIO 未分配 |
| `Vdd` | Pin 15 | `BUCK_OUT`，约 3.3V | 代码按 3.3V 逻辑时序实现，方向匹配 |
| `TM` | Pin 16/17 | `TM1` 热敏电阻采样网络 | 代码只有 mock 温度 |
| `LAT` | Pin 23 | 经过 22R 接 MCU | 有 latch 骨架，但 GPIO 未分配 |
| `DO` | Pin 24 | 未连接 | 代码未使用，单机芯非级联场景可接受 |
| 电机 A/B 相 | Pin 27~30 | 接 DRV8833 `AOUT/BOUT` | 有 DRV8833 骨架，但节奏未按机芯参数闭环 |

说明：原理图里 FPC 符号显示了 31/32 等额外脚位，BOM 中连接器是 `X10A20H30G` 30P，额外脚大概率是封装机械/屏蔽/固定焊盘，最终仍应以实际封装和 FPC 实物方向复核。

### 3.3 电源与功率路径

- `IP2326`：负责 Type-C 输入、2S 锂电池升压充电和 `BAT_STAT` 充电状态输出。
- `TPS563200DDCR`：用 `RFB1=33.2k`、`RBF2=10k` 反馈电阻生成约 3.3V `BUCK_OUT`，匹配 ESP32 和打印头逻辑电源需求。
- `VH_OUT`：由 `G_VH -> N-MOS -> P-MOS` 控制，把 `2S+` 切到打印头 VH。
- `H3`：用于控制 MCU 供电方式，涉及 `BUCK_OUT` 和 `ESP32_3V3`，固件文档和代码还没有把这个硬件选择写清楚。
- `EQD`：由 `2S+ -> 220k/100k 分压 -> 1k/100nF 滤波` 输出到 MCU，是电池电压采样网络。

当前代码中，`G_VH` 的安全逻辑方向与原理图一致：`LEVEL_VH_OFF=0`，高电平打开 N-MOS 后拉低 P-MOS 栅极，从而打开 `VH_OUT`。

### 3.4 缺纸、温度、电池、按键

- 缺纸检测：原理图使用 LM393 比较器，输出 `PAPER_N`。JX-2R-01 规格书说明：有反射时输出低电平，无反射时输出高电平。因此真实驱动应把 `PAPER_N == LOW` 转成 `paperPresent=true`。
- 温度检测：JX-2R-01 规格书给出热敏电阻 `R25=30k`、`B=3950K`；原理图用 `R7=30k` 上拉和 `C10=100nF` 滤波，硬件匹配。当前代码还没有 ADC 采样和 NTC 换算。
- 电池检测：原理图有 `EQD` 分压网络，但当前代码没有 `PIN_EQD`，也没有真实电池 ADC 计算。
- 充电状态：原理图将 IP2326 `BAT_STAT` 接到 MCU；当前代码只有 `ChargeStatus` 抽象，没有真实读取。
- 按键：原理图有 `G_KEY` 输入，当前代码没有按键任务、按键服务或按键事件。

## 4. 已匹配的功能

- 打印几何参数匹配：代码使用 384 点/行、48 字节/行、6 个 STB 组，每组 64 点，符合 JX-2R-01 规格书。
- 最大同时加热点数匹配：代码限制单组不超过 64 dots，符合规格书“同时打印点数 64 dots”的限制。
- 逻辑电压方向匹配：硬件用约 3.3V `BUCK_OUT` 做逻辑电源，代码的 SPI/LAT/STB 设计按 3.3V 逻辑可工作。
- VH 安全要求匹配：规格书要求不打印时关闭 VH、VH 开关过程中 STB 保持非工作；代码中 `setVh(true)` 前检查 STB inactive，异常时关闭 VH/STB。
- STB 控制方向基本匹配：热敏头 STB 为逻辑输入，高电平选通；代码默认 `LEVEL_STB_INACTIVE=0`，active 为 1。
- DRV8833 安全方向匹配：`nSLEEP` 低电平休眠、`nFAULT` 低有效，代码方向与原理图/DRV8833 习惯一致。
- 电机限流硬件匹配：原理图 `R15/R18=560mR`，图中标注 `0.357A`，与 JX-2R-01 规格书相电流 0.357A 对得上。
- PC 侧协议工具与固件方向匹配：当前固件使用 UART 协议，工具可发送 `PING/STATUS/SAFE_OFF/SENSOR_TEST/PRINT_TEST`。

## 5. 未匹配或匹配不完整的功能

### 5.1 GPIO 映射还没有落地

当前最大缺口是 `src/bsp/bsp_pins.h` 仍全部为 `PIN_UNASSIGNED`。这意味着即使开启 `MP_ENABLE_HW_THERMAL_HEAD=1` 或 `MP_ENABLE_HW_STEPPER=1`，真实驱动也无法操作硬件。

需要补全：

- `PIN_PAPER_N`
- `PIN_G_NFAULT`
- `PIN_G_NSLEEP`
- `PIN_G_KEY`
- `PIN_BAT_STAT`
- 新增 `PIN_EQD`
- `PIN_G_VH`
- `PIN_LAT`
- `PIN_DI`
- `PIN_CLK`
- `PIN_AIN1/AIN2/BIN1/BIN2`
- `PIN_TM1`
- `PIN_STB1~PIN_STB6`

### 5.2 `EQD` 电池电压网络缺失

原理图有 `EQD`，但当前：

- `docs/pin_map.md` 没有 `EQD`。
- `src/bsp/bsp_pins.h` 没有 `PIN_EQD`。
- `SensorDriver` 没有真实 ADC 读取。

这是明确未匹配项。后续应把 `EQD` 作为电池电压 ADC 输入，按分压比约 `3.2` 换算：`batteryMv = adcMv * (220k + 100k) / 100k`，再做 ADC 校准。

### 5.3 真实传感器驱动未实现

当前 `SensorDriver` 仍返回 mock：

- `PAPER_N` 未读取，且未实现低有效反相。
- `TM1` 未读取，未按 30k/B3950 NTC 公式换算温度。
- `EQD` 未读取，电池电压仍是 mock。
- `BAT_STAT` 未读取，充电状态仍是 mock/UNKNOWN。
- `G_NFAULT` 虽然在 stepper driver 有读取骨架，但没有 GPIO 分配和系统级真实联调。

### 5.4 打印头热模型还没有按规格书闭环

代码现在有保守脉冲算法，但还没有完整使用规格书中的热模型：

- 规格书推荐 `VH=7.2V`、64 点同时加热时，25℃ 典型 Ton 约 0.49ms。
- 代码默认 `heatPulseStartUs=400us`、`heatPulseMaxUs=800us`，属于骨架参数，还不是校准结果。
- 代码没有根据真实 `VH_OUT`、加热点电阻、导线电阻、环境温度曲线做能量计算。
- 代码没有基于打印速度和行周期实现完整的能量累计/冷却约束。

因此当前可以说“安全边界方向匹配”，不能说“打印浓度和温升模型已经匹配”。

### 5.5 电机步进节奏还没有按机芯规格闭环

JX-2R-01 规格书给出：

- 走纸精度：一个步进距离 0.0625mm。
- 打印点密度：8 dots/mm，即点距 0.125mm。
- 电机有推荐驱动频率和占空比表。

当前代码只在每行后调用一次 `stepper.stepForward()`，还没有明确：

- 一行打印后到底应走 1 步还是 2 步。
- 不同速度下的 PPS、占空比和加减速策略。
- 是否需要按规格书建议使用 PWM 控制电机电流和噪声。
- 正向走纸方向是否与 `AOUT1/AOUT2/BOUT1/BOUT2` 接线方向一致。

所以当前电机驱动只能算“底层动作骨架”，还不是已匹配机芯走纸模型。

### 5.6 按键功能未匹配

硬件有 `G_KEY` 按键输入，当前固件没有：

- 按键 GPIO 初始化。
- 按键去抖。
- 按键事件队列。
- 长按/短按功能定义。

如果产品目标里需要手动走纸、开机、测试页或安全停止，这部分还没有实现。

### 5.7 IP2326 充电管理只匹配了硬件存在性

硬件有 IP2326、Type-C、`BAT_STAT`、LED 和充电参数电阻。当前固件只保留了 `ChargeStatus` 抽象，没有真实逻辑：

- 未读取 `BAT_STAT`。
- 未定义 `BAT_STAT` 高/低电平到 `CHARGING/FULL/FAULT` 的映射。
- 未暴露 USB 输入、充电中、充满或异常状态。
- IP2326 的 NTC、超时、输入过压/欠压等硬件保护由芯片自己处理，固件目前没有可观测性。

### 5.8 DevKit 引脚复用风险未复核

原理图把 `STB1~STB6` 接在 Right 8~13，根据 GPIO 图推断为 GPIO40/39/38/37/36/35。这里有两个风险：

- 映射图上这些脚带 FSPI/JTAG/板载 LED 等复用标记，必须确认当前 DevKitC-1 变体是否可安全作为普通 GPIO。
- `GPIO38` 在图中标注了 `RGB_LED`，如果实际板载 RGB LED 占用该脚，STB3 会受影响。

在没有复核前，不建议直接开启真实热敏头硬件宏。

## 6. 推荐的后续匹配顺序

1. 先补 `docs/pin_map.md`，把上面的 MCU 排针/GPIO 映射写入文档，并新增 `EQD`。
2. 补 `src/bsp/bsp_pins.h`，但仍保持硬件宏默认关闭。
3. 新增真实 `SensorDriver`，先只做输入读取，不触发打印：
   - `PAPER_N` 低有效反相。
   - `TM1` ADC + 30k/B3950 NTC 换算。
   - `EQD` ADC + 3.2 分压换算。
   - `BAT_STAT` 数字输入。
   - `G_NFAULT` 数字输入。
4. 用 `SENSOR_TEST` 在真实串口上验证 paper/temp/battery/charge/fault。
5. 用 `SAFE_OFF` 和万用表确认所有危险输出安全态。
6. 开启 `MP_ENABLE_HW_STEPPER=1`，只做小步数低速电机测试。
7. 开启 `MP_ENABLE_HW_THERMAL_HEAD=1`，先做 `HEAD_SHIFT_TEST` 和 VH 关闭条件下的 `HEAD_STB_TEST` 波形测试。
8. 最后才进入真实加热打印，并用很低黑点密度和短行数验证。

## 7. 当前判断

- 硬件设计与固件架构方向：匹配。
- 打印机芯接口、STB 分组、48 字节行宽、安全关闭思路：匹配。
- 真实 GPIO、真实传感器、真实电池采样、按键、充电状态、电机节奏、热模型：未完全匹配。
- 当前还不能宣称“固件已经适配真实硬件并可直接打印”，只能宣称“固件已经为这套硬件建立了较合理的安全骨架和软件打印链路”。
